# 底仓策略执行口径（以此为准）

这份文档专门用来固定 `list / guadao` 和 `transfer` 的业务含义，避免后续实现时再次理解反。

## 1. 两种策略的目的

### `transfer`

- 目的：赚差价
- 核心判断：`transfer_real_ratio >= 阈值`
- 成本口径：`Steam 买入成本 = Steam 市价 × 0.73`
- 利润来源：`C5 卖出到手` 高于 `Steam 买入成本`

一句话：

- `transfer` 是低成本买 Steam，再高价卖 C5，吃价差

### `list` / `guadao`

- 目的：导余额
- 核心判断：`listing_ratio <= 阈值`
- 核心不是赚差价，而是把库存价值换成 `Steam 余额`

一句话：

- `list` / `guadao` 是卖 Steam、补 C5，用更低损耗把货换成 Steam 余额

## 2. `transfer` 的正确执行含义

`transfer` 不是“买到的那件再拿去 C5 卖”。

正确含义是：

1. 用 `Steam 余额` 在 Steam 市场买入同类饰品
2. 在 C5 上卖出的，是原来底仓里已经有、且当前可交易的那件
3. 新买到的那件，是补回底仓/替换底仓的那件
4. 所以：
   - 卖的不是刚买的那件
   - 刚买的那件会进入冷却
   - 但这不影响把“原来的可交易底仓”立即拿去 C5 卖

## 3. `transfer` 的状态理解

从执行角度看：

1. `Steam` 先买
2. `C5` 立刻卖原底仓可交易资产
3. 卖出完成后，底仓实际由“刚买回来的冷却资产”顶上
4. 等这件 replacement 过冷却后，状态再回到正常 `holding`

也就是说：

- `transfer` 不需要等冷却以后再去挂 C5
- 需要等冷却的，是补回来的 replacement 资产，不是拿去卖的那件

## 4. 自动执行范围

自动执行时，只允许操作你上传了 `STEAM_COOKIES` 的那个 Steam 账号。

具体口径：

- 扫描库存：仍然可以读取 C5 绑定的全部账号，用来做全局判断
- 自动 `Steam` 买卖：只允许使用 `STEAM_COOKIES` 对应账号
- 自动 `C5` 卖出：只允许使用这个账号名下、当前可交易的底仓资产

不能做的事：

- 不能自动卖别的 C5 绑定账号的资产
- 不能把 `transfer` 里刚买到的那件，误当成 C5 卖出对象

## 5. 代码实现时必须满足

### `transfer`

- 先验证当前有“本账号、同品类、可交易”的旧底仓资产可卖
- 再执行 `Steam buy_listing`
- 再把“旧底仓资产”上架到 `C5 sale_create`
- 后续跟踪新买回来的 replacement 资产冷却状态

### `list` / `guadao`

- 继续走原来的：
  - `Steam` 卖
  - `C5` 补仓

## 6. 一句总记忆

- `transfer`：赚差价，买 Steam，卖 C5，卖的是旧底仓
- `list`：导余额，卖 Steam，补 C5
