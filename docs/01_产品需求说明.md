# CS2 理财助手需求规格说明书 v1.3

## 1. 产品目标

产品长期目标是做一个围绕 CS2 饰品交易决策与执行辅助的本地化工具，覆盖以下方向：
1. 单个饰品监控与提醒。
2. 系列、篮子、自定义组合监控。
3. 仓位、空仓、回接计划等本地账本记录。
4. 围绕 `C5` 的交易辅助能力。
5. 基于库存的做T收益率分析与提醒。
6. 提醒、推送与后续自动化能力。

当前 `v1.0` 只聚焦一个核心模块：**基于全部已绑定 Steam 账号库存，辅助筛选适合做T的饰品类型，并输出做T收益率结果与提醒**。

当前收敛后的落地重点如下：
1. 基于全库存聚合做T候选。
2. 计算做T收益率并支持 `top / bottom` 输出。
3. 输出对应 `steamId` 与 `nickname`，便于定位库存来源。
4. 提醒模块独立拆分，只保留“高收益快报 + 15:30 固定提醒”。

## 2. 核心使用场景

### 2.1 单个饰品监控

用户选择一个具体饰品后，系统持续跟踪其价格、在售数量、求购价、涨跌幅，并支持：
1. 低于某个价格时提醒买入。
2. 高于某个价格时提醒卖出。
3. 下跌达到某个百分比时提醒。
4. 上涨达到某个百分比时提醒。
5. 同一个饰品设置多条提醒条件。

### 2.2 系列或篮子监控

用户希望监控一整个武器箱、印花系列、刀系、热门系列，或者自定义一篮子饰品，把它们当成“指数”观察。

系统需要支持两类指数：
1. 平台已有指数或系列。
2. 用户自定义篮子指数。

### 2.3 持仓与空仓记录

用户不只是看行情，还要知道自己现在是：
1. 持仓中。
2. 已卖出，空仓观察中。
3. 卖飞后等待回接。
4. 纯观察，没有持仓。

系统需要本地记录每个标的的状态、成本、仓位、预期回接价、目标卖出价、已实现收益和未实现收益。

### 2.4 C5 交易辅助

用户主要使用 `C5` 交易，因此系统优先支持：
1. 查询当前最低价和可买库存。
2. 找到符合条件的在售列表。
3. 触发一键成交或快速购买。
4. 查看当前在售并做批量改价。
5. 在条件不满足时下架或停止自动操作。

第一阶段交易执行规则：
1. 命中买入条件后，系统只提醒，不自动下单。
2. 命中卖出条件后，系统只提醒，不自动卖出。
3. 购买和卖出都需要用户确认。

### 2.5 基于库存的做T收益率模块

这是 `v1.0` 的核心模块，也是当前全项目优先级最高的功能模块。

用户当前的实际使用方式已经明确为：固定按 `0.73` 的 Steam 余额折扣口径，在 `Steam` 买入、在 `C5` 卖出，计算做T收益率。

第一阶段确认的基础公式：

`折算比 = C5饰品价格 ÷ (0.869 × Steam饰品最低在售价)`

`做T收益率 = 折算比 × 0.869 - 0.73`

等价写法：

`做T收益率 = (C5饰品价格 ÷ Steam饰品最低在售价) - 0.73`

模块实现要求：
1. 汇总所有已绑定 `Steam` 账号的库存。
2. 同类型饰品按 `marketHashName` 去重归并。
3. 每个饰品类型计算折算比和做T收益率。
4. 支持按收益率从高到低 `top N` 输出，也支持从低到高 `bottom N` 输出。
5. 支持附加过滤条件，如 `C5` 最低售价大于等于指定阈值。
6. 如果存在 `C5` 价格但缺少 `Steam` 价格的饰品，必须单独记录并提示排查。

统一输出字段：
1. 饰品名称。
2. `marketHashName`。
3. 对应 `steamId`。
4. 对应 `nickname`。
5. 做T收益率（百分比）。
6. 折算比。
7. `C5` 最低售价。
8. `Steam` 最低售价。

## 3. 功能模块

### 3.1 标的库

系统需要维护统一的标的主表，至少包含：
1. 中文名。
2. `marketHashName`。
3. 平台 `itemId`。
4. 分类信息。
5. 所属系列。

### 3.2 库存做T模块

这是 `v1.0` 的主模块，其他模块优先级都低于它。

模块主能力：
1. 从所有已绑定 `Steam` 账号拉取库存。
2. 聚合同类型饰品。
3. 计算折算比与做T收益率。
4. 输出高收益 `top N` 候选。
5. 输出低收益 `bottom N` 候选。
6. 标注所属账号和库存状态。

### 3.3 价格采集

系统需要定时采集以下维度：
1. C5 最低在售价。
2. C5 在售数量。
3. C5 求购最高价。
4. Steam 价格或 Steam 近 7 天均价。
5. `CSQAQ` 的 Steam 价格补充、系列、指数、排行等数据。
6. 用户所有已绑定 `Steam` 账号的库存。
7. 按 `marketHashName` 聚合后的库存饰品类型池。

价格源优先级说明：
1. 库存做T模块中的 `C5` 价格优先使用库存查询结果自带价格。
2. `Steam` 价格优先用批量接口获取。
3. 如果存在 `C5` 价格但 `Steam` 价格缺失，系统必须记录问题并提示。

### 3.4 提醒引擎

系统支持通用提醒规则：
1. 绝对价格提醒。
2. 涨跌幅提醒。
3. 比例提醒。
4. 仓位状态联动提醒。

此外，提醒引擎还需要补充一条独立于通用规则引擎的“做T汇总提醒”能力：
1. 检查 `C5` 全部绑定账号全部库存。
2. 计算做T收益率。
3. 按 `Top N + C5价格 > xx` 过滤。
4. 当前只保留“高收益快报 + 15:30 固定提醒”两种提醒。

提醒规则要支持：
1. 一个标的配置多条规则。
2. 买入提醒与卖出提醒分开配置。
3. 冷却时间，避免反复触发。
4. 一次触发后保留历史记录。

提醒模块拆分要求：
1. 价格采集与计算放在市场策略模块。
2. 规则判定放在提醒规则模块。
3. `ServerChan` 等通知渠道放在通知模块。
4. 做T汇总提醒单独做服务层，后续便于加更多提醒类型。

### 3.5 仓位账本

API 无法完整表达真实交易成本和“卖飞后计划接回”的主观策略，因此系统必须有本地仓位账本。

成本口径说明：
1. 第一阶段由用户手工记录成本。
2. 系统以手工录入结果为准，不自动推导真实成本。

建议账本字段：
1. 标的。
2. 状态。
3. 持仓数量。
4. 持仓成本。
5. 计划买回价。
6. 计划卖出价。
7. 已实现收益。
8. 未实现收益。
9. 备注。

### 3.6 交易执行层

执行层以 `C5` 为主，优先支持：
1. 快速购买。
2. 条件筛选后购买。
3. 查看在售。
4. 改价。
5. 下架。

第一阶段建议采用“提醒 + 人工确认 + 一键执行”的模式，不建议一上来做纯无人值守自动下单。

### 3.7 通知中心

后续应支持多个通知渠道：
1. 桌面提醒。
2. Telegram。
3. 企业微信或微信转发。
4. 邮件。

第一阶段已确认使用 `企业微信 ServerChan`。

## 4. 业务规则建议

### 4.1 指数规则

指数分成两种：
1. 官方指数或热门系列指数，直接使用平台提供的数据。
2. 自定义指数，按用户手选饰品做平均价、加权价或总市值指数。

第一阶段自定义指数口径已确认：
1. 指数值 = 一篮子饰品当前价格直接相加。
2. 第一阶段不做平均价和权重计算。

### 4.2 提醒判定规则

建议提醒条件支持这些写法：
1. `价格 <= 100` 时提醒买入。
2. `价格 >= 130` 时提醒卖出。
3. `24h 跌幅 <= -6%` 时提醒。
4. `从最近高点回撤 >= 10%` 时提醒。
5. `比例 <= 0.88` 或 `>= 0.96` 时提醒。
6. `做T收益率 >= 6%` 时提醒优先关注。

### 4.3 仓位状态

建议使用以下状态：
1. `holding`
2. `watch_only`
3. `waiting_reentry`
4. `sold_out`
5. `disabled`

### 4.4 风控规则

自动执行前至少要检查：
1. C5 余额是否足够。
2. 目标价格是否仍然满足条件。
3. 该提醒是否处于冷却中。
4. 是否超过单笔和单日预算。
5. 是否需要自动发货条件。

补充说明：
1. 第一阶段所有买入和卖出动作都需要用户确认。
2. 第一阶段不启用无人值守自动交易。

## 5. 分阶段落地建议

### 第一阶段（v1.0）

`v1.0` 专攻库存做T模块，其他功能只保留必要配套：
1. 所有已绑定 `Steam` 账号库存汇总。
2. 同类型饰品聚合。
3. 基于库存的做T收益率 `top / bottom` 输出。
4. 基于 `Top N + C5价格门槛` 的高收益快报。
5. 每天 `15:30` 的固定提醒。
6. 缺少 `Steam` 价格的异常记录与查看命令。
7. 必要的价格采集、提醒和一键执行入口。

### 第二阶段

扩展到组合和策略层：
1. 系列指数监控。
2. 自定义篮子指数。
3. 基于库存与趋势的联动筛选。

### 第三阶段

再做自动化与策略强化：
1. 自动扫货规则。
2. 自动挂售改价。
3. 订单回放和盈亏分析。
4. 更完整的风控系统。

## 6. 当前结论

这个产品当前最适合定义成“以库存做T收益率分析和提醒为核心，以 `C5` 为交易执行核心，以 `SteamDT + CSQAQ` 为价格补充”的工具，而不是一开始就做成大而全的行情站或量化机器人。

`v1.0` 最重要的不是把所有接口都接完，而是先跑通这条链路：

`全账号库存汇总 -> 同类型聚合 -> 价格采集 -> 折算比 / 做T收益率计算 -> Top/Bottom 输出 -> 提醒 -> C5 手动执行`

## 7. 最近已落地补充（做T收益率与提醒模块）

### 7.1 当前主线入口

当前主入口固定为：

```text
python .\main.py t-profit scan
python .\main.py t-profit missing-steam
python .\main.py notify t-profit ...
```

其中：
1. `t-profit scan` 负责做T扫描与输出。
2. `t-profit missing-steam` 负责查看最近一次缺失 Steam 价格的明细。
3. `notify t-profit` 负责提醒配置、单次执行和常驻轮询。

### 7.2 当前做T扫描实现

最近已落地的扫描逻辑包括：
1. 汇总全部 C5 绑定 Steam 账号库存。
2. 按 `marketHashName` 聚合同类型饰品。
3. 记录每个饰品类型的总数量、冷却数量、不冷却数量。
4. 计算折算比和做T收益率。
5. 支持按做T收益率从高到低或从低到高排序输出。
6. 支持按 `C5 最低售价门槛` 过滤。
7. 支持按库存状态筛选后再查询价格。

### 7.3 当前库存状态定义

当前代码里已落地的库存状态规则为：
1. `all`：不做库存状态筛选，全部都参与。
2. `tradable_only`：该饰品类型在当前库存里全部是不冷却。
3. `cooldown_only`：该饰品类型在当前库存里全部是冷却中。
4. `mixed_only`：该饰品类型在当前库存里同时存在冷却和不冷却。

补充说明：
1. `all` 是筛选模式，不是单独的一种库存状态。
2. 输出结果里需要带库存状态摘要，例如“1 冷却 / 2 不冷却”。
3. 价格查询前先做库存状态过滤，以减少无效价格请求。
4. 当前扫描命令对外只暴露三种筛选口径：`all`、`all_cooldown`、`has_tradable`。

### 7.4 当前价格源执行策略

当前实际执行策略固定为：

```text
C5(库存 + C5价) + CSQAQ(Steam主价) + SteamDT(Steam兜底)
```

细化如下：
1. `C5` 库存接口自带价格优先作为 C5 价格。
2. 如果库存没有价格，再用 C5 批量价格接口补齐。
3. `CSQAQ` 作为 Steam 最低售价主来源。
4. `SteamDT` 作为 Steam 最低售价兜底。
5. 如果出现“有 C5 价格但没有 Steam 价格”，该饰品不进入做T候选，但必须记录并提示。

### 7.5 缺失 Steam 价格处理

当前缺价处理规则已落地为：
1. 写入 `data/c5_t_yield_missing_steam_prices.json`。
2. 在扫描输出中提示缺价数量。
3. 在提醒输出中也提示缺价数量。
4. 用户可通过 `python .\main.py t-profit missing-steam` 查看明细。

### 7.6 当前提醒模块结构

最近已落地的提醒模块结构为：
1. 扫描层：库存汇总、价格获取、收益率计算。
2. 输出层：主脚本直接输出做T结果。
3. 提醒层：判断本次是否需要提醒。
4. 推送层：本地输出或 ServerChan 发送。

### 7.7 当前提醒配置项

当前做T提醒配置至少包含：
1. `top_n`
2. `min_price`
3. `steam_discount`
4. `hot_threshold_pct`
5. `poll_interval_minutes`
6. `daily_summary_time`
7. `inventory_scope`
8. `allow_cached_fallback`
9. `cache_max_age_minutes`

### 7.8 当前提醒规则

当前已落地的提醒规则固定为两类：
1. 高收益快报。
2. `15:30` 固定提醒。

库存范围只分三种：
1. `all`：全部。
2. `all_cooldown`：全冷却。
3. `not_all_cooldown`：存在不冷却。

提醒规则：
1. 轮询间隔按配置执行。
2. 如果存在做T收益率 `>= hot_threshold_pct` 的候选，则触发高收益快报。
3. 如果高收益候选和上次完全相同，则不重复推送，只在本地输出。
4. 每天 `15:30` 之后当天首次轮询时，固定触发一次提醒。
5. `15:30` 固定提醒与高收益快报共用同一套库存范围配置。

### 7.9 当前提醒输出与降级规则

当前提醒模块的实际行为是：
1. 未命中推送条件时，只在本地输出。
2. 命中推送条件但未配置 ServerChan 时，也降级为本地输出。
3. 命中推送条件且已配置 ServerChan 时，发送推送。
4. 提醒脚本停止后，提醒也随之停止。

### 7.10 当前缓存规则

当前缓存策略已落地为：
1. 默认优先实时拉取全部库存。
2. 实时拉取失败时，允许回退到指定时长内的缓存。
3. 缓存只做失败兜底，不作为主数据源。
4. 提醒和扫描共用一份库存缓存。
